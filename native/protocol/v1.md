# IPC Protocol v1

## Transport

- shell starts engine with `QProcess`
- shell writes UTF-8 JSON lines to engine `stdin`
- engine writes UTF-8 JSON lines to `stdout`
- engine logs diagnostic text to `stderr`

Every transport message must be serialized as one compact JSON object per line.

## Message Types

### Request

```json
{ "type": "request", "id": "1", "method": "engine.ping", "params": {} }
```

### Response

Successful response:

```json
{ "type": "response", "id": "1", "ok": true, "result": { "engineVersion": "0.1.0" } }
```

Error response:

```json
{ "type": "response", "id": "1", "ok": false, "error": { "code": "UNKNOWN_METHOD", "message": "Unsupported method" } }
```

### Event

```json
{ "type": "event", "event": "engine.ready", "payload": { "protocol": "1", "engineVersion": "0.1.0" } }
```

Structured startup failure event:

```json
{
  "type": "event",
  "event": "engine.startupFailed",
  "payload": {
    "stage": "bootstrap",
    "code": "BOOTSTRAP_FAILED",
    "message": "...",
    "paths": { "logFilePath": "/logs/engine.log" }
  }
}
```

## Startup Handshake

1. engine emits `engine.ready`
2. shell sends `health.snapshot`
3. shell sends `app.snapshot`
4. shell decides whether the startup target is dashboard or commissioning from `app.snapshot`
5. shell opens the operator surface only after both snapshots succeed
6. shell fails startup if any startup step does not complete inside its watchdog window

In the current scaffold the shell still renders the runtime scaffold immediately, but the protocol and shell state machine now model this startup order explicitly.

Future slices will add domain methods such as:

- `projects.list`
- `projects.create`
- `lights.snapshot`
- `audio.snapshot`

## Startup Lifecycle States

The shell startup state machine currently uses these lifecycle states:

- `idle`
- `launching-process`
- `waiting-for-ready-event`
- `waiting-for-health-snapshot`
- `waiting-for-app-snapshot`
- `ready`
- `failed`

Contract rules:

- `engine.ready` means the engine process is alive, storage bootstrap completed, and the protocol version is compatible enough to continue startup.
- `health.snapshot` provides operational readiness and diagnostics metadata.
- `app.snapshot` provides engine-owned UI-routing state such as commissioning status and startup target surface.
- `planning.changed` is a non-startup domain event used to tell the shell that planning state mutated and snapshots should be refreshed.
- `ready` means the shell has received both health and app snapshots, not merely that the process exists.

## Rules

- `id` is required for requests and responses
- `event` is required for events
- all payloads must be JSON objects
- unknown methods return structured errors, not crashes
- protocol version mismatch should surface as `engine.ready` incompatibility or a structured startup error
- shell provides runtime directories through environment variables before launch:
  - `SSE_APP_DATA_DIR`
  - `SSE_LOG_DIR`
  - `SSE_PROTOCOL_VERSION`
- `health.snapshot` should expose storage metadata needed for startup diagnostics:
  - schema version
  - journal mode
  - integrity-check result
- `engine.ready` and `health.snapshot` should expose the engine log file path for recovery tooling
- bootstrap failures should emit `engine.startupFailed` with:
  - `stage`
  - `code`
  - `message`
  - planned runtime `paths`
- `app.snapshot` currently returns:
  - `runtime.protocol`
  - `runtime.engineVersion`
  - `runtime.paths`
  - `shell.workspace`
  - `shell.window`
  - `commissioning.hasCompletedSetup`
  - `commissioning.stage`
  - `commissioning.hardwareProfile`
  - `planning.viewFilter`
  - `planning.sortBy`
  - `planning.dashboardView`
  - `planning.deckMode`
  - `planning.selectedProjectId`
  - `planning.selectedTaskId`
  - `startup.targetSurface`
  - `startup.operatorUiAllowed`
- `planning.snapshot` currently returns:
  - `projects`
  - `tasks`
  - `activityLog`
  - `settings.viewFilter`
  - `settings.sortBy`
  - `settings.dashboardView`
  - `settings.deckMode`
  - `settings.selectedProjectId`
  - `settings.selectedTaskId`
  - `counts.projectCount`
  - `counts.taskCount`
  - `counts.runningTaskCount`
  - `counts.completedTaskCount`
- `planning.context` currently returns:
  - `selectedProject`
  - `projectIndex`
  - `projectCount`
  - `settings.viewFilter`
  - `settings.sortBy`
  - `settings.dashboardView`
  - `settings.deckMode`
  - `settings.selectedProjectId`
  - `settings.selectedTaskId`
  - `selectedTaskId`
  - `selectedTask`
  - `taskIndex`
  - `tasks`
  - `taskCount`
  - `runningTask`
- `planning.settings.update` currently accepts:
  - `viewFilter`: one of `all`, `todo`, `in-progress`, `blocked`, `done`
  - `sortBy`: one of `manual`, `priority`, `date`, `name`
  - `dashboardView`: one of `kanban`, `lighting`, `audio`
  - `deckMode`: one of `project`, `light`, `audio`
  - `selectedProjectId`: string or `null`
  - `selectedTaskId`: string or `null`
  - selecting a project without an explicit task restores that project's first task by native ordering
  - selecting a task infers its owning project and keeps project/task selection consistent
- `planning.select` currently accepts exactly one of:
  - `projectId`
  - `taskId`
  - `projectDirection`: `next` or `prev`
  - `taskDirection`: `next` or `prev`
  - project cycling follows native project order and restores the first task in the target project
  - task cycling stays within the currently selected project
- `planning.task.timer` currently accepts:
  - `taskId`: required string
  - `action`: one of `start`, `stop`, `toggle`
  - successful responses include:
    - `resolvedAction`
    - `task`
    - `context`
  - the current native behavior matches the Electron route:
    - `toggle` resolves from the current running state
    - explicit `start` resets `lastStarted` to now
    - explicit `stop` accumulates elapsed seconds from `lastStarted`
- `planning.task.toggleComplete` currently accepts:
  - `taskId`: required string
  - successful responses include:
    - `task`
    - `context`
- successful planning mutations currently emit `planning.changed` with:
  - `reason`
  - `projectId`
  - `taskId`
- `settings.update` currently accepts:
  - `workspace`: one of `planning`, `lighting`, `audio`, `setup`
  - `window.width`: integer between `800` and `8192`
  - `window.height`: integer between `600` and `4320`
  - `window.maximized`: boolean
- `storage.importLegacyDb` accepts:
  - `path`: required string path to a legacy `db.json`
  - `force`: optional boolean, default `false`; required when replacing existing native planning data

Startup import behavior:

- when native planning tables are empty, the engine will attempt a safe one-way auto-import on startup
- import source priority:
  - `SSE_LEGACY_DB_PATH` if set
  - repo-local `data/db.json` when running from the repo in development
- set `SSE_DISABLE_AUTO_IMPORT=1` to disable startup auto-import
- auto-import never replaces existing native planning data
- retry of a failed or skipped import remains explicit through `storage.importLegacyDb`

Successful `settings.get` / `settings.update` responses now include:

- raw persisted `settings`
- structured `shell.workspace`
- structured `shell.window.width`
- structured `shell.window.height`
- structured `shell.window.maximized`

Successful `storage.importLegacyDb` responses now include:

- `sourcePath`
- `sourceSchemaVersion`
- `replacedExistingData`
- `importedProjects`
- `importedTasks`
- `importedChecklistItems`
- `importedActivityEntries`
- `normalizedRunningTasks`
- `updatedSettings`

## Ownership

- The shell owns process supervision, retry behavior, and desktop presentation.
- The engine owns persisted application state, commissioning state, and UI-routing decisions derived from persisted state.
- QML does not issue raw protocol messages directly. All transport calls go through shell-side adapter methods.
- The shell may cache snapshot data for rendering, but the engine remains the source of truth.

## Versioning

- Additive changes inside existing result objects are allowed in protocol v1 if old clients can ignore the new fields safely.
- Removing fields, changing field types, or changing startup sequencing requires a protocol version bump.
- A new startup-critical method required for shell boot must be documented in the startup handshake and treated as part of the versioned contract.

## Error Taxonomy

Structured errors and startup failures should use stable codes grouped by responsibility:

- startup:
  - `BOOTSTRAP_FAILED`
  - `STARTUP_TIMEOUT`
  - `HEALTH_SNAPSHOT_FAILED`
  - `APP_SNAPSHOT_FAILED`
- process supervision:
  - `PROCESS_ERROR`
  - `PROCESS_EXIT`
  - `ENGINE_BINARY_MISSING`
  - `ENGINE_NOT_RUNNING`
- protocol:
  - `UNKNOWN_METHOD`
  - `INVALID_PARAMS`
  - `INVALID_ENGINE_MESSAGE`
- persistence:
  - `STORAGE_ERROR`
  - `RUNTIME_DIRECTORY_ERROR`
- import:
  - `IMPORT_SOURCE_NOT_FOUND`
  - `IMPORT_FAILED`
  - `IMPORT_REQUIRES_FORCE`

Codes may appear either in engine error responses or shell-generated failure surfaces. The meaning of a code must remain stable once released.

## Design Notes

- JSON Lines is chosen for v1 because it is easy to debug and cross-platform over `QProcess`
- if binary framing is needed later, preserve the message schema and swap only the framing layer
- the shell should treat `engine.ready` as "process is alive and protocol-compatible", not as "system is fully healthy"
- the engine should bootstrap storage paths before emitting `engine.ready`, so first-launch storage failures are caught before the shell proceeds
- `app.snapshot` is the first non-shell UI-driving contract and exists specifically to keep commissioning/dashboard routing out of ad hoc shell state
